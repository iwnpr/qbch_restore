# qbch_request_restore — описание проекта

## 1. Задача

На сервере из-за чрезмерной нагрузки часть запросов от сервиса **qbch_api** к сервису
**qbch_db_saver_sdc** была потеряна.

Схема работы в норме:
- qbch_api кладёт данные запроса в **Redis** (хэш по ключу `QBCH:dlrequest:<TransactionId>`)
  и отправляет в **Kafka** событие — строку-ключ этого же хэша;
- qbch_db_saver_sdc читает событие из Kafka, забирает данные из Redis и сохраняет в БД.

Что пошло не так: TTL ключей в Redis был **8 часов**, а saver под нагрузкой разгрёб очередь
более чем через 8 часов. К моменту обработки часть ключей уже удалилась из Redis по TTL —
saver получал событие, но данных в Redis уже не было. В его логах это видно как:

```
Не удалось найти ключ в кеше "QBCH:dlrequest:88abf198-1d54-4260-83e9-05904ba04051"
```

Такие запросы не попали в БД и считаются потерянными.

**Цель:** заново собрать данные потерянных запросов из логов Serilog сервиса qbch_api,
положить их обратно в Redis и повторно отправить событие в Kafka — чтобы saver их дочитал
и сохранил в БД.

## 2. Исходные данные

| Источник | Где | Что берём |
|---|---|---|
| Логи **qbch_db_saver_sdc** | `logs_saver/` | строки `Не удалось найти ключ в кеше "QBCH:dlrequest:<guid>"` → список потерянных ключей |
| Логи **qbch_api** | `logs/` | запросный XML и связку TransactionId↔RequestId |

### Цепочка связывания

Потерянный ключ из логов saver содержит `TransactionId` (это и есть `dlrequest`).
Запросный XML в логах qbch_api содержит `ИдентификаторЗапроса` (это `RequestId`).
Связать одно с другим позволяет пара соседних строк обработчика qbch_api:

```
пропавший ключ  QBCH:dlrequest:<TransactionId>                      (логи saver)
   │
   │  QBCHProcessingHandlerV3.Handle начало: TransactionId="<guid>"  ┐
   │  QBCHProcessingHandlerV3: RequestId=<guid>                      ┘  связка TransactionId ↔ RequestId
   │
   └─ Qbch_db_lib...RepositoryV3 XML для поиска субъектов (fip_search_for_api_3_0): <?xml ...>
        <ЗапросСведений ... ИдентификаторЗапроса="<RequestId>"> ... </ЗапросСведений>
```

Связывание устойчиво к многопоточности логов: TransactionId из строки «Handle начало»
привязывается к **следующей** строке `RequestId` (между ними иногда вклинивается лог БД,
но порядок сохраняется — проверено на всех парах sample-лога).

## 3. Решение

Консольная утилита (.NET 8). Работа разбита на 3 идемпотентных этапа; промежуточные данные
хранятся в локальной БД **SQLite** (`restore.db`).

```
dotnet run -- stage1            # логи saver  -> потерянные ключи (missing.txt + таблица missing)
dotnet run -- stage2            # логи qbch_api -> связки + запросный XML (таблицы links, requests)
dotnet run -- stage3            # свод и отчёт (dry-run, БЕЗ записи)
dotnet run -- stage3 --push     # запись хэшей в Redis + события в Kafka
```

Запись в Redis и Kafka воспроизводит поведение боевого qbch_api 1:1:
- формат хэша — как в `KeyValueStorageService.AddHashArray` / `QBCHProcessingCompleteHandlerV3`;
- событие — `Message<Null, string>` со значением-ключом, `Acks.All`, как в `KafkaService.Produce`.

### Структура проекта

```
Program.cs                       Точка входа, разбор этапов
AppConfig.cs                     Настройки из appsettings.json
appsettings.json                 Пути логов, подключения Redis/Kafka

Data/RestoreDb.cs                SQLite: схема и операции
  таблицы: missing, links, requests, restore

Stages/Stage1Missing.cs          Логи saver → потерянные ключи
Stages/Stage2ParseApiLogs.cs     Логи qbch_api → связки и запросный XML
Stages/Stage3BuildAndPush.cs     Свод, dry-run / запись

Redis/RedisWriter.cs             Сборка хэша + HashSet + TTL
Kafka/KafkaSender.cs             Produce события
```

### Таблицы SQLite

| Таблица | Поля | Назначение |
|---|---|---|
| `missing` | transaction_id | потерянные ключи (этап 1) |
| `links` | transaction_id, request_id, request_date_time | связка (этап 2) |
| `requests` | request_id, request_xml | запросный XML (этап 2) |
| `restore` | transaction_id, request_id, status, note, pushed_at | журнал (этап 3): matched / no_link / no_xml / pushed / error |

## 4. Что восстанавливается

Только **запросная часть** dlrequest — то, что присутствует в логах qbch_api:

| Поле хэша Redis | Источник |
|---|---|
| `request_xml` | XML `fip_search_for_api_3_0` из лога |
| `response_guid` | TransactionId |
| `request_id` | ИдентификаторЗапроса |
| `request_date_time` | строка `... QBCHIIIController "<guid>" Request: ...` |
| `request_certificate_thumbprint` | `-` (отпечаток в логах не пишется) |
| `error_code` | `0` |
| `api_version`, `contract_version` | `3.0` |

После записи saver (`RepositoryV3.CreateDlRequestV3`) сохранит сам dlrequest, субъектов,
запросы и пользователей из `request_xml`.

## 5. Ключевые решения и ограничения

- **Только запрос.** Подписанный ответ ЦБ (`response_signed_data`, `response_xml`) и разбивка
  по КБКИ-задачам (подключи `:<ОГРН>`) в логах отсутствуют — восстановить их нельзя. Поля ответа
  в БД останутся пустыми.
- **`error_code = 0`.** saver не сохраняет запись, если `error_code == 12` и нет поля
  `qbch_tasks_end_date_time` (считает её «ожидающей результат» и перекидывает в error-топик).
  Поэтому ставим `0`.
- **`AbonentId` будет пустым.** saver сопоставляет абонента по отпечатку сертификата, а отпечаток
  в логах qbch_api не пишется → `request_certificate_thumbprint = "-"`.
- **Маршрутизация V3.** saver уходит в ветку V3, т.к. `request_xml` содержит `Версия="3."`
  (и дополнительно ставим `api_version`/`contract_version = "3.0"`).
- **Стек:** C# .NET 8 + Microsoft.Data.Sqlite, StackExchange.Redis 2.8.16, Confluent.Kafka 2.6.1
  (те же клиенты, что у боевых сервисов — сериализация полей совпадает).
- **Безопасность:** stage3 по умолчанию работает в режиме **dry-run** и ничего не пишет.
  Реальная запись в боевые Redis/Kafka — только при явном `--push`.

## 6. Настройки (`appsettings.json`)

```jsonc
"Paths":  { "ApiLogsDir": "logs", "SaverLogsDir": "logs_saver", "MissingTxt": "missing.txt", "Db": "restore.db" }
"Redis":  { "ConnectionString": "...", "DbIndex": 0, "TtlMinutes": 480 }   // DbIndex и адрес — как у боевого сервиса
"Kafka":  { "BootstrapServers": "...", "Topic": "QBCHMessagesTopic_V3_27" } // тот же топик, что читает saver
```

## 7. Порядок применения

1. Положить реальные логи saver в `logs_saver/`, логи qbch_api в `logs/`.
2. `stage1` → проверить `missing.txt` / число потерянных ключей.
3. `stage2` → распарсить логи qbch_api.
4. `stage3` (dry-run) → изучить отчёт: сколько восстановимо, сколько без связки / без XML.
5. Заполнить боевые адреса Redis/Kafka в `appsettings.json`.
6. `stage3 --push` → запись. Результат по каждому ключу — в таблице `restore`.
